19 // This is essentially a wrapper around NSURLConnection for POSTs and GETs.
20 // If setPostData: is called, then POST is assumed.
21 //
22 // When would you use this instead of NSURLConnection?
23 //
24 // - When you just want the result from a GET or POST
25 // - When you want the "standard" behavior for connections (redirection handling
26 // an so on)
27 // - When you want to avoid cookie collisions with Safari and other applications
28 // - When you want to provide if-modified-since headers
29 // - When you need to set a credential for the http
30 // - When you want to avoid changing WebKit's cookies
31 //
32 // This is assumed to be a one-shot fetch request; don't reuse the object
33 // for a second fetch.
34 //
35 // The fetcher may be created auto-released, in which case it will release
36 // itself after the fetch completion callback. The fetcher
37 // is implicitly retained as long as a connection is pending.
38 //
39 // But if you may need to cancel the fetcher, allocate it with initWithRequest:
40 // and have the delegate release the fetcher in the callbacks.
41 //
42 // Sample usage:
43 //
44 // NSURLRequest *request = [NSURLRequest requestWithURL:myURL];
45 // GTMHTTPFetcher* myFetcher = [GTMHTTPFetcher httpFetcherWithRequest:request];
46 //
47 // [myFetcher setPostData:[postString dataUsingEncoding:NSUTF8StringEncoding]]; // for POSTs
48 //
49 // [myFetcher setCredential:[NSURLCredential authCredentialWithUsername:@"foo"
50 // password:@"bar"]]; // optional http credential
51 //
52 // [myFetcher setFetchHistory:myMutableDictionary]; // optional, for persisting modified-dates
53 //
54 // [myFetcher beginFetchWithDelegate:self
55 // didFinishSelector:@selector(myFetcher:finishedWithData:)
56 // didFailSelector:@selector(myFetcher:failedWithError:)];
57 //
58 // Upon fetch completion, the callback selectors are invoked; they should have
59 // these signatures (you can use any callback method names you want so long as
60 // the signatures match these):
61 //
62 // - (void)myFetcher:(GTMHTTPFetcher *)fetcher finishedWithData:(NSData *)retrievedData;
63 // - (void)myFetcher:(GTMHTTPFetcher *)fetcher failedWithError:(NSError *)error;
64 //
65 // NOTE: Fetches may retrieve data from the server even though the server
66 // returned an error. The failWithError selector is called when the server
67 // status is >= 300 (along with any server-supplied data, usually
68 // some html explaining the error).
69 // Status codes are at <http://www.w3.org/Protocols/rfc2616/rfc2616-sec10.html>
70 //
71 //
72 // Proxies:
73 //
74 // Proxy handling is invisible so long as the system has a valid credential in
75 // the keychain, which is normally true (else most NSURL-based apps would have
76 // difficulty.) But when there is a proxy authetication error, the the fetcher
77 // will call the failedWithError: method with the NSURLChallenge in the error's
78 // userInfo. The error method can get the challenge info like this:
79 //
80 // NSURLAuthenticationChallenge *challenge
81 // = [[error userInfo] objectForKey:kGTMHTTPFetcherErrorChallengeKey];
82 // BOOL isProxyChallenge = [[challenge protectionSpace] isProxy];
83 //
84 // If a proxy error occurs, you can ask the user for the proxy username/password
85 // and call fetcher's setProxyCredential: to provide those for the
86 // next attempt to fetch.
87 //
88 //
89 // Cookies:
90 //
91 // There are three supported mechanisms for remembering cookies between fetches.
92 //
93 // By default, GTMHTTPFetcher uses a mutable array held statically to track
94 // cookies for all instantiated fetchers. This avoids server cookies being set
95 // by servers for the application from interfering with Safari cookie settings,
96 // and vice versa. The fetcher cookies are lost when the application quits.
97 //
98 // To rely instead on WebKit's global NSHTTPCookieStorage, call
99 // setCookieStorageMethod: with kGTMHTTPFetcherCookieStorageMethodSystemDefault.
100 //
101 // If you provide a fetch history (such as for periodic checks, described
102 // below) then the cookie storage mechanism is set to use the fetch
103 // history rather than the static storage.
104 //
105 //
106 // Fetching for periodic checks:
107 //
108 // The fetcher object can track "Last-modified" dates on returned data and
109 // provide an "If-modified-since" header. This allows the server to save
110 // bandwidth by providing a "Nothing changed" status message instead of response
111 // data.
112 //
113 // To get this behavior, provide a persistent mutable dictionary to setFetchHistory:,
114 // and look for the failedWithError: callback with code 304
115 // (kGTMHTTPFetcherStatusNotModified) like this:
116 //
117 // - (void)myFetcher:(GTMHTTPFetcher *)fetcher failedWithError:(NSError *)error {
118 // if ([[error domain] isEqual:kGTMHTTPFetcherStatusDomain] &&
119 // ([error code] == kGTMHTTPFetcherStatusNotModified)) {
120 // // [[error userInfo] objectForKey:kGTMHTTPFetcherStatusDataKey] is
121 // // empty; use the data from the previous finishedWithData: for this URL
122 // } else {
123 // // handle other server status code
124 // }
125 // }
126 //
127 // The fetchHistory mutable dictionary should be maintained by the client between
128 // fetches and given to each fetcher intended to have the If-modified-since header
129 // or the same cookie storage.
130 //
131 //
132 // Monitoring received data
133 //
134 // The optional received data selector should have the signature
135 //
136 // - (void)myFetcher:(GTMHTTPFetcher *)fetcher receivedData:(NSData *)dataReceivedSoFar;
137 //
138 // The bytes received so far are [dataReceivedSoFar length]. This number may go down
139 // if a redirect causes the download to begin again from a new server.
140 // If supplied by the server, the anticipated total download size is available as
141 // [[myFetcher response] expectedContentLength] (may be -1 for unknown
142 // download sizes.)
143 //
144 //
145 // Automatic retrying of fetches
146 //
147 // The fetcher can optionally create a timer and reattempt certain kinds of
148 // fetch failures (status codes 408, request timeout; 503, service unavailable;
149 // 504, gateway timeout; networking errors NSURLErrorTimedOut and
150 // NSURLErrorNetworkConnectionLost.) The user may set a retry selector to
151 // customize the type of errors which will be retried.
152 //
153 // Retries are done in an exponential-backoff fashion (that is, after 1 second,
154 // 2, 4, 8, and so on.)
155 //
156 // Enabling automatic retries looks like this:
157 // [myFetcher setIsRetryEnabled:YES];
158 //
159 // With retries enabled, the success or failure callbacks are called only
160 // when no more retries will be attempted. Calling the fetcher's stopFetching
161 // method will terminate the retry timer, without the finished or failure
162 // selectors being invoked.
163 //
164 // Optionally, the client may set the maximum retry interval:
165 // [myFetcher setMaxRetryInterval:60.]; // in seconds; default is 600 seconds
166 //
167 // Also optionally, the client may provide a callback selector to determine
168 // if a status code or other error should be retried.
169 // [myFetcher setRetrySelector:@selector(myFetcher:willRetry:forError:)];
170 //
171 // If set, the retry selector should have the signature:
172 // -(BOOL)fetcher:(GTMHTTPFetcher *)fetcher willRetry:(BOOL)suggestedWillRetry forError:(NSError *)error
173 // and return YES to set the retry timer or NO to fail without additional
174 // fetch attempts.
175 //
176 // The retry method may return the |suggestedWillRetry| argument to get the
177 // default retry behavior. Server status codes are present in the error
178 // argument, and have the domain kGTMHTTPFetcherStatusDomain. The user's method
179 // may look something like this:
180 //
181 // -(BOOL)myFetcher:(GTMHTTPFetcher *)fetcher willRetry:(BOOL)suggestedWillRetry forError:(NSError *)error {
182 //
183 // // perhaps examine [error domain] and [error code], or [fetcher retryCount]
184 // //
185 // // return YES to start the retry timer, NO to proceed to the failure
186 // // callback, or |suggestedWillRetry| to get default behavior for the
187 // // current error domain and code values.
188 // return suggestedWillRetry;
189 // }

386 // Fetch History is a useful, but a little complex at times...
387 //
388 // The caller should provide a mutable dictionary that can be used for storing
389 // Last-Modified-Since checks and cookie storage (setFetchHistory implicity
390 // calls setCookieStorageMethod w/
391 // kGTMHTTPFetcherCookieStorageMethodFetchHistory if you passed a dictionary,
392 // kGTMHTTPFetcherCookieStorageMethodStatic if you passed nil.
393 //
394 // The caller can hold onto the dictionary to reuse the modification dates and
395 // cookies across multiple fetcher instances.
396 //
397 // With a fetch history dictionary setup, the http fetcher cache has the
398 // modification dates returned by the servers cached, and future fetches will
399 // return 304 to indicate the data hasn't changed since then (the data in the
400 // NSError object will be of length zero to show nothing was fetched). This
401 // reduces load on the server when the response data has not changed. See
402 // shouldCacheDatedData below for additional 304 support.
403 //
404 // Side effect: setFetchHistory: implicitly calls setCookieStorageMethod:
405 - (NSMutableDictionary *)fetchHistory;
406 - (void)setFetchHistory:(NSMutableDictionary *)fetchHistory;
407 
408 // For fetched data with a last-modified date, the fetcher can optionally cache
409 // the response data in the fetch history and return cached data instead of a
410 // 304 error. Set this to NO if you want to manually handle last-modified and
411 // status 304 (Not changed) rather than be delivered cached data from previous
412 // fetches. Default is NO. When a cache result is returned, the didFinish
413 // selector is called with the data, and [fetcher status] returns 200.
414 //
415 // If the caller has already provided a fetchHistory dictionary, they can also
416 // enable fetcher handling of 304 (not changed) status responses. By setting
417 // shouldCacheDatedData to YES, the fetcher will save any response that has a
418 // last modifed reply header into the fetchHistory. Then any future fetches
419 // using that same fetchHistory will automatically load the cached response and
420 // return it to the caller (with a status of 200) in place of the 304 server
421 // reply.

423 - (void)setShouldCacheDatedData:(BOOL)flag;

452 // GTM HTTP Logging
453 //
454 // All traffic using GTMHTTPFetcher can be easily logged. Call
455 //
456 // [GTMHTTPFetcher setIsLoggingEnabled:YES];
457 //
458 // to begin generating log files.
459 //
460 // Log files are put into a folder on the desktop called "GTMHTTPDebugLogs"
461 // unless another directory is specified with +setLoggingDirectory.
462 //
463 // Each run of an application gets a separate set of log files. An html
464 // file is generated to simplify browsing the run's http transactions.
465 // The html file includes javascript links for inline viewing of uploaded
466 // and downloaded data.
467 //
468 // A symlink is created in the logs folder to simplify finding the html file
469 // for the latest run of the application; the symlink is called
470 //
471 // AppName_http_log_newest.html
472 //
473 // For better viewing of XML logs, use Camino or Firefox rather than Safari.
474 //
475 // Projects may define GTM_HTTPFETCHER_ENABLE_LOGGING to 0 to remove all of the
476 // logging code (it defaults to 1). By default, any data uploaded via PUT/POST
477 // w/ and NSInputStream will not be logged. You can enable this logging by
478 // defining GTM_HTTPFETCHER_ENABLE_INPUTSTREAM_LOGGING to 1 (it defaults to 0).
